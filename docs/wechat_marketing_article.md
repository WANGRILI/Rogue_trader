# AI Agent 会怎样进入量化研究？我用 RogueTrader 做了一次实验

最近我在整理一个开源项目：RogueTrader。

GitHub 地址：

https://github.com/WANGRILI/Rogue_trader

它是一个面向加密货币和传统金融资产的多智能体 LLM 研究框架。简单说，就是让一组 AI Agent 像一个研究团队一样协作：有人看技术面，有人看新闻，有人看链上数据，有人负责多空辩论，有人负责风控，最后形成一份可以追踪和复盘的交易研究结果。

但这篇文章我不想只写“项目功能介绍”。

真正做下来以后，我越来越觉得，AI Agent 进入量化研究，重点可能不只是“让模型预测涨跌”。更重要的是：它会反过来改变我们组织数据、记录过程、设计研究系统的方式。

## 不是所有量化场景都适合 Agent

先说一个比较实际的感受：高频交易里，Agent 的存在感可能没有想象中那么强。

高频场景讲究的是延迟、撮合、微观结构、执行质量、风控限速和基础设施稳定性。这里很多问题不是大模型擅长的。你不能让一个 LLM 在毫秒级订单簿里慢慢推理，也不能把执行路径交给一个不可控的自然语言系统。

如果讨论的是高频策略，我会更倾向于把 Agent 放在外围：

- 帮助研究员解释异常交易日。
- 自动总结回测表现。
- 检查参数漂移。
- 归纳日志中的异常模式。
- 辅助生成研究报告。

它更像研究助理，而不是交易核心。

但中长线不太一样。

中长线研究天然需要处理更多非结构化信息：宏观环境、行业叙事、资金流向、链上行为、新闻事件、风险情景、仓位逻辑。这些东西很难完全写成一个固定公式，但又不能只凭感觉。

这正是 Agent 比较有价值的地方。

它可以把“研究员脑子里的一套分析流程”显性化：先看数据，再找证据，再让多空观点互相挑战，最后由风控视角重新审查。这件事未必让结论立刻变得神奇，但会让研究过程更完整、更容易复盘。

## 加密资产不应该只看 K 线

我最初改造 RogueTrader，是因为传统金融分析框架直接拿来分析加密货币，总觉得缺了一块。

加密市场不是只看 K 线。

BTC、ETH 以及其他加密资产背后，有稳定币供应、交易所筹码集中度、DeFi TVL、链上活跃度、矿工行为、永续合约资金费率、市场情绪和宏观流动性。很多时候，价格只是最后呈现出来的结果。

所以 RogueTrader 里专门加入了链上分析师。

它会关注这些问题：

- 资金是在流入还是流出？
- 稳定币供应是否在扩张？
- DeFi TVL 是否在迁移？
- 交易所成交是否高度集中？
- 永续合约资金费率是否说明多头过度拥挤？
- 恐惧贪婪指数是否进入极端区间？
- BTC 是否接近历史上常被观察的周期风险区域？

项目里目前整合了 CoinGecko、DeFiLlama、Blockchain.com、Alternative.me、yfinance 等数据源，也包含 NVT、Pi Cycle、Funding Rate、CME Gap、Fear & Greed 等指标。

这些指标不能保证预测市场，但能让分析不再只有“涨了所以强、跌了所以弱”。

## 多智能体不是热闹，而是分工

RogueTrader 不是单轮问答，而是把一次研究拆成多个角色。

第一层是分析师团队：

- 市场分析师：看价格、成交量和技术指标。
- 社交情绪分析师：看社区情绪和叙事变化。
- 新闻分析师：看宏观事件、行业新闻和外部冲击。
- 基本面分析师：面向股票或传统资产时分析财务与估值。
- 链上分析师：专门处理加密资产的链上和衍生品指标。

第二层是研究团队：

- 多头研究员负责论证看涨逻辑。
- 空头研究员负责识别风险和下行情景。
- 研究经理负责综合多空观点。

第三层是交易员和风控：

- 交易员把研究结果整理成交易计划。
- 风控团队从激进、保守、中立三个视角重新审查。
- 投资组合经理给出最终决策。

这套结构的重点，不是让 AI 说得更热闹，而是强制它把不同视角分开。

单个模型很容易顺着上文走。输入材料偏乐观，它就更容易乐观；输入材料偏悲观，它就更容易悲观。

多智能体辩论至少能让系统主动生成反方观点，再由上层角色做综合。

这对加密资产尤其重要。因为同一个市场里，经常同时存在完全相反的叙事：一边是 ETF、长期持有者、生态增长和宏观流动性；另一边是杠杆拥挤、监管风险、流动性抽离和鲸鱼砸盘。

## 让小白用户改 Agent，而不是改代码

这次我比较重视的一个改动，是把 Agent 的模型路由和身份配置集中到了一个 YAML 文件里：

```text
configs/agents.yaml
```

以前如果想改某个 Agent 的提示词身份、关注重点、模型选择，通常要进 Python 代码里找对应文件。对熟悉项目的人问题不大，但对刚 clone 下来的用户并不友好。

现在可以在一个配置文件里调整每个 Agent：

- 使用 quick 模型还是 deep 模型。
- 是否单独指定某个模型。
- 这个 Agent 的身份是什么。
- 它应该重点关注什么。
- 它的输出风格应该更交易化、研究化还是风控化。

这不是为了把所有 Prompt 都暴露出去。

我反而觉得，Agent 配置应该有边界：身份、关注点和表达风格适合开放给用户调整；数据上下文、工具调用规则、状态字段和最终输出约束，仍然应该由代码维护。

原因很简单：小白用户需要能调试不同模型和不同 Agent 设定，但系统本身也需要稳定边界。否则改 Prompt 改到最后，工具调用和状态流都可能失控。

这个配置文件的价值在于，它让用户可以更容易做 A/B 测试：

- 市场分析师用轻量模型是否够用？
- 研究经理换成更强推理模型是否会更稳？
- 链上分析师的关注重点改成“资金流和操纵风险”后，报告是否更有用？
- 风控 Agent 更保守或更激进时，最终结论会怎样变化？

多智能体系统真正有趣的地方，不是“一个模型厉害不厉害”，而是不同角色、不同模型、不同提示设定组合起来以后，系统行为会发生什么变化。

## 输出结构比想象中重要

这是我改造过程中一个很强的体会。

过去很多量化系统的数据格式，是给程序读的，不是给 Agent 读的。

比如一个策略回测结果可能只需要几个字段：收益率、夏普、最大回撤、胜率、换手率。程序看这些够了，研究员也能看懂。

但 Agent 要真正参与研究，它需要更多上下文：

- 这份数据对应哪个交易标的？
- 分析日期是什么？
- 数据是否严格截断到当时？
- 使用的是哪个数据源？
- 这次运行用了哪些配置？
- 哪些报告参与了最终决策？
- 终端日志、分段报告和完整状态是否能对应到同一次运行？
- 这个结论以后能不能被评估、比较和复盘？

这就是为什么我后来专门整理了 RogueTrader 的输出目录。

现在一次标准运行会写入一个独立目录：

```text
my_results/运行结果/<时间戳>_<标的>/
```

里面包含：

- `运行索引.json`
- `报告.md`
- `状态.json`
- `最终决策.json`
- `运行配置.json`
- `分段报告/`
- `终端日志.log`

这个改动看起来不大，但它是 Agent 进入量化工作流的基础。

因为 Agent 不只是在“生成一段话”，它会变成系统里的一个节点。节点之间要协作，就必须有清晰的输入、输出、时间戳、配置和可追踪状态。

我甚至觉得，未来很多量化项目如果想适配 Agent，第一步不是接入大模型，而是整理自己的数据和结果格式。

数据要有时间边界。

结果要能被机器读。

报告要能追溯到数据源。

决策要能被后续评估系统消费。

这可能比写一个漂亮的 Prompt 更重要。

## 一次真实运行应该是可复盘的

最近我还修了一个很小但很真实的问题：日志和结果不要分散。

以前很多脚本习惯这样跑：

```bash
python xxx.py 2>&1 | tee result.log
```

这当然能保存终端输出，但它有一个问题：这个 `.log` 文件和结构化结果目录之间没有天然绑定。跑多了以后，你很容易分不清哪个 log 对应哪个报告。

现在 RogueTrader 更推荐直接运行：

```bash
uv run --frozen python my_scripts/roguetrader1.py
```

它会把这次运行的终端日志自动写入同一个运行目录里的 `终端日志.log`。如果中途异常或手动中断，也会尽量写出当前已完成的部分结果，并标记为未完成。

这类细节不炫，但很关键。

一个研究系统如果不能复盘，就很难迭代。尤其是多智能体系统，过程本身往往和最终结论一样重要：哪个 Agent 调了什么工具、哪个报告提供了关键证据、辩论在哪里发生转向、最终风控为什么接受或否决交易计划，这些都应该能留下来。

## 怎么和已有量化项目衔接？

我不认为 RogueTrader 应该替代已有量化系统。

更合理的方式，是把它当成研究层或解释层，接在已有流程旁边。

我现在对 LLM 和 Agent 接入量化项目的理解，大概可以分成几个层次。

最表层的是工具层，比如让模型帮忙解释报错、生成回测摘要、写一点数据处理代码。

再往里一点，是传统量化项目比较容易想到的方向：参数寻优、策略库整理、因子库检索、回测结果归因。这些事情很实用，但本质上还是把 LLM 当成一个更聪明的助手，围绕已有系统做增强。

还有一层更有意思：把投资团队的理念和协作流程模拟出来。比如一个团队里本来就会有人偏宏观、有人偏技术面、有人偏风险控制、有人专门挑战乐观假设。过去这些东西往往存在于团队经验和会议讨论里，很难变成系统的一部分。

RogueTrader 在我看来，更接近这一层。

它不是一个完整的基金投研系统，也不是一个自动下单引擎。它更像是“投资团队理念的中间层”的轻量化实现：把分析师、研究员、交易员、风控和组合经理这些角色拆开，让它们围绕同一份数据和同一个标的形成一套可记录、可复盘、可调整的研究过程。

这个定位对我来说很重要。

因为它既没有把 Agent 神化成“直接替代策略”，也没有把 Agent 降级成“只会写代码和调参数的工具”。它尝试站在中间：把原本比较隐性的投研流程，变成一个可以被工程化调用的流程节点。

一个比较现实的衔接方式是：

1. 现有量化系统继续负责数据采集、清洗、因子计算、回测和风控约束。
2. RogueTrader 读取已经整理好的本地数据和策略结果。
3. Agent 对某个标的、某个日期、某次信号或某段回测结果生成解释。
4. 最终输出 JSON 和 Markdown 报告。
5. 这些结果再回到原量化系统，用于归档、评估、复盘或人工审阅。

也就是说，Agent 不一定要直接下单。

它可以先回答几个更稳妥的问题：

- 为什么这个信号在这一天出现？
- 当前信号和链上资金状态是否一致？
- 多头和空头分别能找到哪些证据？
- 这个交易计划最大的失效条件是什么？
- 这次分析和过去相似场景有什么不同？

这些问题不要求 Agent 做毫秒级决策，但能补足传统量化系统里常常缺失的“解释层”。

## 中长线研究更容易先 Agent 化

如果按时间尺度看，我会更看好中长线研究先适配 Agent。

原因不是中长线更容易赚钱，而是它更适合自然语言、结构化报告和多源信息综合。

中长线分析里，很多判断并不是一个因子直接决定的，而是多个证据共同构成：

- 趋势是否成立？
- 链上资金是否支持？
- 宏观环境是否友好？
- 市场情绪是否过热？
- 杠杆是否拥挤？
- 风险收益比是否合理？

这些问题天然适合拆成 Agent 流程。

高频系统追求的是执行效率，中长线系统更需要解释、记录、复盘和风控讨论。Agent 在后者里的价值会更早显现。

这也是我为什么把 RogueTrader 的重点放在链上分析、多智能体辩论、Agent 配置、结构化输出和本地数据上，而不是把它包装成一个自动交易机器人。

下一步我比较想优化的是记忆层。

现在项目里已经有 Agent 记忆系统的雏形，但更理想的状态是：每个 Agent 都有相对独立的记忆库。市场分析师记住市场结构，链上分析师记住链上异常和资金流，风控 Agent 记住历史上的失效条件，组合经理记住不同风险场景下的决策后果。

如果后续接入 OpenViking 这类可挂载的记忆/知识库系统，可以考虑给每个 Agent 单独挂载自己的记忆空间。这样它们不是共享一团模糊上下文，而是像真实团队成员一样，各自积累自己的经验，再在一次新的分析中拿出来参与讨论。

这会让“投资团队理念的中间层”更进一步：不只是角色分工，还能逐步沉淀角色经验。

## 默认 DeepSeek，降低实验成本

RogueTrader 支持多个 LLM Provider，包括 DeepSeek、OpenAI、Anthropic、Google、xAI、OpenRouter 和 Ollama。

默认使用 DeepSeek。

原因也很现实：多智能体系统调用次数多，如果每一步都用高价模型，个人开发者很难长期试验。

当前默认配置把常规 Agent 任务放在 `deepseek-v4-flash`，把研究经理、投资组合经理这类更需要综合判断的节点放在 `deepseek-v4-pro`。如果你想用其他模型，也可以切换 Provider，或者接入本地 Ollama。

## 这个项目现在能做什么？

目前 RogueTrader 已经具备几个基础能力：

- 可以通过 CLI 或 Python API 启动多智能体分析。
- 可以选择市场、社交、新闻、基本面、链上等分析师。
- 可以通过 `configs/agents.yaml` 调整 Agent 身份、关注点和模型路由。
- 可以输出完整报告、分段报告、终端日志和完整状态。
- 可以提取最终 BUY / HOLD / SELL 等基础结构化决策。
- 可以将一次运行写入独立目录，保留配置、状态和结果。
- 可以使用本地 processed/parquet 数据，并按分析日期截断，减少未来函数风险。

它仍然是研究框架，不是成熟交易系统。

但对我来说，这个阶段已经够有价值：它验证了一个方向，即 Agent 可以先作为量化研究的解释层、讨论层和复盘层进入系统，而不是一上来就接管交易执行。

## 适合谁关注？

如果你是以下几类人，可能会对 RogueTrader 感兴趣：

- 想研究 AI Agent 在金融分析中如何落地的开发者。
- 想把 LLM 接入量化研究流程的工程师。
- 关注加密资产链上数据和资金结构的研究者。
- 正在整理本地数据、回测结果和信号评估系统的人。
- 想找一个可以二次开发的多智能体交易研究框架的人。

你可以把它看成一个实验底座。

可以继续加链上数据源，比如 Glassnode、Dune、Arkham。

可以把本地 point-in-time 数据做得更严格。

可以把最终决策 JSON 变得更强约束、更适合评估。

可以把每个 Agent 的记忆库独立挂载到 OpenViking，形成更清晰的角色经验沉淀。

也可以让它专门分析已有量化策略的信号，而不是直接分析交易标的。

## 最后

AI 做交易分析，真正困难的地方不只是“让模型说出一个方向”。

更困难的是：让它有分工、有证据、有反方、有风控、有记录，并且能被复盘。

我现在越来越觉得，未来量化项目适配 Agent 的过程，可能不会从“让 AI 直接交易”开始，而是从更朴素的地方开始：

把数据整理得更清楚。

把运行过程记录得更完整。

把自然语言判断变成可以复盘的结构化结果。

把研究员脑子里的经验，逐步沉淀成可调用、可评估、可迭代的 Agent 流程。

RogueTrader 目前还在早期，但它已经把这个方向跑通了一个可用的开源版本。

免责声明：RogueTrader 是研究和分析框架，不是自动实盘系统，也不构成任何投资建议。加密资产波动剧烈，请独立判断并控制风险。
